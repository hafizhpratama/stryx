# `flow/redirect-open`

> Catches untrusted request input flowing to a redirect call as the target URL.

## Metadata

| Field | Value |
|---|---|
| Rule ID | `flow/redirect-open` |
| Status | experimental |
| Severity | high |
| Frameworks | nextjs >= 13, express, fastify (single-file + cross-file slice 2) |
| Default | enabled |
| Added in | v0.1.0 |

## What this rule catches

Open-redirect happens when an application redirects a browser to a
URL the attacker controls. The classic exploit: a phishing email
links to `https://your-app.com/login?return=https://phishing.com`;
the user trusts the domain, logs in, then your app redirects them
to the phishing page where they re-enter their credentials thinking
they're still on your site. The trust transfer is the whole point.

Stryx flags when a value sourced from the request body or query
parameters reaches `NextResponse.redirect(...)`, the `redirect(...)`
helper from `next/navigation`, or `res.redirect(...)` /
`Response.redirect(...)` shapes as the target URL â€” without passing
through a recognised URL allow-list along the way.

## Why AI gets this wrong

Cursor / Claude Code / Copilot routinely generate OAuth callback
handlers, "return-to" redirects, and "share-link" handlers that
pass the body-supplied URL straight to a redirect. The pattern
`const { redirectTo } = await req.json(); return NextResponse
.redirect(redirectTo);` is what AI tools produce when asked to
"build a Next.js login redirect handler" â€” the runtime threat model
("the attacker can pass any URL, the browser will trust your domain
during the redirect") never appears in the prompt.

## Bad example (real AI output)

```ts
// Generated by Claude Code, 2026-05.
// Prompt: "Add a /api/return endpoint that redirects the user back
//          to the URL they came from after successful authentication."

import { NextResponse } from "next/server";
import type { NextRequest } from "next/server";

export async function POST(req: NextRequest) {
  const { returnTo } = await req.json();
  return NextResponse.redirect(returnTo);
}
```

## Good example

```ts
import { NextResponse } from "next/server";
import type { NextRequest } from "next/server";

const ALLOWED_HOSTS = new Set(["myapp.com", "app.myapp.com"]);

export async function POST(req: NextRequest) {
  const { returnTo } = await req.json();
  let parsed: URL;
  try {
    parsed = new URL(returnTo);
  } catch {
    return NextResponse.json({ error: "Invalid URL" }, { status: 400 });
  }
  if (!ALLOWED_HOSTS.has(parsed.host)) {
    return NextResponse.json({ error: "Host not allowed" }, { status: 403 });
  }
  return NextResponse.redirect(returnTo);
}
```

## Taint signature

| Field | Value |
|---|---|
| Source labels | `UserInput` (body / query / headers) |
| Sink ids | `http.redirect` (Next.js `NextResponse.redirect`, `next/navigation` `redirect`, Express `res.redirect`, `Response.redirect`) |
| Sanitizers recognized | URL-allowlist check (`Set.has` / `Array.includes` on `parsed.host` etc., validator-function form) â€” shared with `flow/ssrf-via-fetch` |
| Scope | `SingleFile` + `CrossFile` |

## Detection logic

1. Walk the program looking for redirect-call sinks â€”
   `NextResponse.redirect(url)`, bare `redirect(url)` from
   `next/navigation`, `res.redirect(url)`, or
   `Response.redirect(url)`.
2. For each sink call, the URL argument is examined for body-source
   taint via the existing `BodySource` step and the structural-
   propagator-walked taint set.
3. If the URL argument is tainted and no recognised allow-list
   sanitiser fired along the path (same allow-list shapes as
   `flow/ssrf-via-fetch`: `Set.has` / `Array.includes` on `.host`
   / `.hostname` / `.origin`, validator-function form), emit a
   Finding at the sink span.
4. **Cross-file (slice 2).** The extract pass simulates each
   exported function with one parameter pre-tainted and records
   `ParamFlow::reaches_redirect_sink_unsanitized` when the
   simulation observes a redirect sink. The run pass walks call
   sites; when a tainted argument flows into a reach-flagged
   parameter slot of a callee resolved via the project index, a
   High-severity finding is emitted at the call site.

## Known false positive zones

- **Server-internal redirects** to a hard-coded path string that
  happens to be a variable holding only literals
  â†’ not a real FP; the tainted-value check ensures we only fire
    on body-tainted URLs.
- **Allow-listed redirects** where the URL is validated via a host
  check but Stryx doesn't yet recognise the validator shape
  â†’ suppress per-line; the rule's allow-list recognition matches
    `flow/ssrf-via-fetch`'s.
- **Login-success redirects to a single hard-coded path** like
  `redirect("/dashboard")`
  â†’ no body taint flows in, no finding.

## LLM escalation prompt (Layer 3)

Not applicable â€” slice 1 is fully deterministic at the AST layer.

## Performance characteristics

- AST analysis: ~0.3ms per file (single-file slice 1)
- Sharing the URL-allow-list sanitiser with `flow/ssrf-via-fetch`
  means no extra step dispatch on the validator path.
